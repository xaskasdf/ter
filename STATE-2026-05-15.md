## Logros arquitectónicos

### 1. TVMAC = 0 demostrado en hardware (Phase B)

`cuda/ter_cuda_bitnet_addonly.cu` — kernel que reemplaza `__dp4a` con conditional add/sub/skip. Para BitNet ternary weights {-1, 0, +1} no necesita multiplicación.

- 5.90 ms/forward Llama 1B M=1 vs v11 7.29 ms → 1.23× más rápido
- 1.15× sobre cuBLAS INT8 TC
- **0 invocaciones de __dp4a** (architectural ideal demonstrated, not just analytical)

### 2. Memory compression escala (Phase C)

`cuda/ter_cuda_packed_scale.cu` — bench a 1B, 8B, 70B (synthetic).

- 4× memory compression holds en TODAS las escalas
- BUT wall-clock advantage NO generaliza linealmente:
  - 1B M=1: 1.90× faster
  - 8B M=1: 0.82× (lose)
  - 70B M=1: 0.56× (lose)
- Per-shape: small-N projections (Wk, Wv, Wo) actually mejoran (Wk 70B = 13.6×)
- FFN-expand shapes son donde TC dominates

### 3. INT4 TC + GEMM-tiled (Phase E)

`cuda/ter_cuda_int4_tc.cu` + `cuda/ter_cuda_int4_tc_tiled.cu`

- INT4 TC peak ~568 TOPS vs INT8 TC ~284 TOPS
- Ternary {-1, 0, +1} packs en int4 (1 bit wasted)
- M=16 across 1B/8B/70B: 1.34-1.97× sobre cuBLAS INT8 TC
- M=64 FFN-expand: GEMM-tiled mejora 6.3× vs naive INT4
- Pero cuBLAS INT8 sigue ganando en M=64 Wgate específicamente

### 4. Hybrid dispatch (Phase E v3)

`cuda/ter_cuda_hybrid_dispatch.cu` — per-shape best-of-toolkit selector.

| Model | M | All-cuBLAS | HYBRID | Speedup |
|---|---:|---:|---:|---:|
| Llama 1B | 1 | 3.11 ms | 2.00 ms | 1.56× |
| Llama 1B | 16 | 4.60 ms | 2.27 ms | 2.02× |
| Llama 8B | 1 | 17.15 ms | 13.09 ms | 1.31× |
| Llama 8B | 16 | 21.75 ms | 13.00 ms | 1.67× |

### 5. End-to-end inference engineering (Sec 6.1 paper)

Tres fixes architecturales que llevaron el end-to-end de 14.7 a 421 t/s en Llama 1B:

1. **Device-pointer scale**: int8 quant scale leído device-side por mm kernel (eliminó 65 D2H syncs/token)
2. **v11 kernel upgrade**: `cuda/ter_cuda_forward_packed.cu` ahora usa el warp-coop v11 kernel (no el viejo v4)
3. **cudaGraph capture**: forward entero (340+ kernel launches) como un graph reusable; toda state en device pointers (pos, token_id)

### 6. BitNet 2B-4T integration (Step 3)

`tools/convert_bitnet_to_packed.py` + `cuda/ter_cuda_forward_bitnet.cu`

- Python converter para i2_s GGUF → col-major packed-trit binary
- C++ binary loader en CUDA forward
- sub_norms support (attn_sub_norm + ffn_sub_norm, BitNet-specific)
- fp16 lm_head via weight-tied token_embd
- 195 t/s end-to-end con real weights

**Tres bugs reverse-engineered de microsoft/BitNet source:**

1. **Code mapping**: microsoft usa `(int8 + 2).astype(uint8)` → codes {0:-1, 1:0, 2:+1}. Empirical histogram confirmó code 1 dominante 50% = zero.
2. **Memory layout**: WEIGHT_PARALLEL packing es `(N//4, K)` con 4 rows per byte (no 4 cols).
3. **Bit order**: row offset 0 en HIGH bits, row offset 3 en LOW bits.

Resultado: hidden state post-output_norm con mean=0.0001, stddev=0.10, range [-0.34, +0.40] — exactamente lo esperado para output transformer pre-vocab.

## 🎉 RESOLVED: fp16 overflow in relu²_mul was the BUG

**Bug found and fixed** (commit e126c6e):

Per-layer diff revealed CUDA's L=0 FFN contribution = 0 (despite Python contributing std 23.25). Root cause: `relu²(gate) * up` overflows fp16 (gate²·up reaches 270k > 65504 max). The inf propagates through `ffn_sub_norm` → rms inf → rms_inv 0 → output 0 → FFN contributes nothing.

**Fix applied**:
1. `s.ff` allocated as `float*` (was `__half*`) — fp32 prevents overflow
2. New `rmsnorm_f32_k` reads fp32, writes fp16 — preserves precision
3. `add_k` clamps to ±65504 — prevents residual stream inf accumulation

**Result**:
- Per-layer hidden state std now matches Python (L=0: 23.12 vs 23.25)
- Top-1 logit at BOS=128000: **token 11 (`,`)** ← EXACT MATCH microsoft reference
- Top-3 logits exact match (`,`, ` and`, ` in`)
- 6/8 tokens match in greedy generation from BOS
- Remaining 25% mismatch is fp16 precision noise in adjacent-token argmax (15=`0` vs 16=`1`)

**Throughput cost**: 195 → 109 t/s (fp32 FFN buffer overhead). Optimizable.

To get FULL bit-exact: convert more buffers to fp32 mid-layer or use bfloat16 (better range than fp16).

  - Total: potential 10-50× energy efficiency vs commercial GPUs
  - At parity wall-clock, this is the architectural argument's core
