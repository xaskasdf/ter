# RTX 3090 production baselines (F12)

**Date:** 2026-05-10
**Host:** flyingWhales (Windows 11 Pro), AMD Ryzen 7 5800X (8c/16t, AVX2, no AVX-512), 48 GB RAM, NVIDIA RTX 3090 24 GB
**Software:** llama.cpp master @ 2e97c5f96, MSVC 14.50 + Ninja + nvcc CUDA 13.2, llama-bench

## Llama 3.2 1B Q8_0

| Backend | tg32 (gen tps) | tg128 (gen tps) | pp128 (prompt tps) |
|---|---:|---:|---:|
| Ter sim, Mac i9-9880H + AVX2 (`test_llama_smoke`) | 0.039 | 0.039 | (host eval) |
| llama.cpp, 5800X CPU only (`-ngl 0`) | 17.24 ± 0.77 | 17.50 ± 0.43 | 81.79 ± 1.78 |
| llama.cpp, RTX 3090 (`-ngl 99`) | 59.91 ± 6.98 | 43.45 ± 15.21 | 2250.94 ± 343.05 |

**Sim vs RTX 3090 gap:** `~1115x slower in tokens/sec at the same model and quant.`

Op count is exact and substrate-independent (the simulator measures TVMACs faithfully). Wall-clock gap reflects: (a) the simulator runs ternary operations on a binary host with per-instruction interpreter overhead, and (b) the analytical cost-per-op for ternary CMOS vs binary GPU is the missing input. Closing the wall-clock gap requires either real ternary silicon (out of scope) or FPGA per-op energy measurements (next experimental step).

## BitNet b1.58 2B-4T i2_s

**Status:** deferred. Stock llama.cpp rejects the i2_s GGUF (ggml type 36 = `TYPE_IQ4_NL_4_4` REMOVED in upstream; microsoft's i2_s is fork-only). Building microsoft/BitNet's bitnet.cpp on flyingWhales hit two infrastructure walls:

1. `setup_env.py` invokes `cmake -T ClangCL` (VS ClangCL toolset, requires VS IDE workload, not present in BuildTools-only install)
2. `utils/codegen_tl2.py` has no `bitnet_b1_58-2B-4T` entry in `ModelShapeDict`; only the older `large` (1.5B), `3B`, and `Llama3-8B-1.58-100B-tokens` variants are supported. Adding the 2B-4T shape `[[2560,2560],[640,2560],[6912,2560],[2560,6912]]` requires picking `BM/BK/bm` tile parameters without tuning guidance.

For the paper, the BitNet result that the simulator already produces stands on its own:
- **End-to-end forward on real BitNet 2B-4T weights** (`test_bitnet_forward`): 18 min, 128256 finite logits, sub_norm wired, attn_sub_norm + ffn_sub_norm absorbing the per-tensor scale (paper sec 3.2)
- **TVMAC eliminated analytically** (`test_bitnet_post_quant`): TinyStories layer matmuls 121856 → 0 TVMAC under bitnet quantization (H3 confirmed)
- The remaining bitnet.cpp baseline would just give a tokens/sec datapoint, not a different architectural conclusion.

## CUDA microbench (the real test)

`cuda/ter_cuda_bench.cu` ports `mm_row` to a CUDA kernel that mirrors the AVX2
algorithm exactly (k-outer, j-inner, X-broadcast, int64 accumulate, scale at
end). Drives the Llama 3.2 1B Q8_0 matmul fabric shapes (Q/K/V/O/gate/up/down
across 16 layers + lm_head).

| Backend | ms / forward (matmul fabric only) | GMAC/s |
|---|---:|---:|
| Sim, Mac i9 + AVX2 scalar | 25,600 | 38 |
| **Sim, RTX 3090, naive CUDA kernel** | **26.06** | 47.5 |
| **Sim, RTX 3090, cuBLAS sgemm fp32** | **6.87** | 180 |

Per-shape detail (cuBLAS sgemm, RTX 3090):

| shape | K x N | GMAC/s |
|---|---:|---:|
| Wgate / Wup (FFN expand) | 2048 x 8192 | ~200 |
| Wq / Wo (attn proj) | 2048 x 2048 | ~183 |
| Wdown (FFN contract) | 8192 x 2048 | ~176 |
| lm_head | 2048 x 128256 | ~210 |
| Wk / Wv (GQA) | 2048 x 512 | ~50 (small, kernel overhead dominant) |

**The 1115x gap from the llama.cpp comparison was a simulation artifact, not
an architectural one.** Once the same algorithm runs on GPU silicon, the
ternary substrate is competitive with production inference: 6.87 ms for the
matmul fabric of one Llama 1B forward, vs ~23 ms total per token from
llama.cpp on the same RTX 3090.

This is a paper-relevant result on its own:
- Op count parity (1.002x lane-MACs vs fp16) holds at the architectural level
- Wall-clock parity holds when the substrate runs on competent hardware
- The remaining headroom is 285 TOPS INT8 tensor cores (Ampere) -- mapping the
  ternary matmul to `cublasLtMatmul` INT8 is the next step (target: > 1 TMAC/s
  for the matmul fabric, ~28x further speedup)

## What still needs FPGA / silicon

Energy per op in joules. The microbench shows wall-clock parity but doesn't
isolate ternary CMOS cell energy from binary CMOS cell energy. An FPGA
prototype with a ternary ALU vs binary ALU baseline would give the missing
J/op factor that, multiplied by the simulator's exact op counts, yields the
system-level energy comparison.

## INT8 tensor cores (the H3 paper headline)

`cuda/ter_cuda_bench2.cu` adds a third backend: `cublasGemmEx` with
`CUDA_R_8I` inputs, `CUDA_R_32I` accumulator, and `CUBLAS_GEMM_DEFAULT_TENSOR_OP`
to enable Ampere INT8 tensor cores (~285 TOPS peak on the 3090).
Activations clipped to BitNet-style ternary {-1, 0, +1}; weights same.
Three batch sizes M = {1, 16, 64} per shape.

Llama 1B forward equivalent (matmul fabric only):

| M | naive ms | sgemm ms | INT8 TC ms | sgemm GMAC/s | INT8 TC GMAC/s |
|--:|---:|---:|---:|---:|---:|
| 1 | 57.3 | 16.17 | **6.55** | 76.6 | **189.2** |
| 16 | (n/a) | 18.88 | **11.77** | 1049.1 | **1683.6** |
| 64 | (n/a) | 22.93 | **11.50** | 3455.1 | **6891.6** |

Per-shape peak (M=64 INT8 TC):
- `lm_head` (2048 x 128256): **13,859 GMAC/s ~= 14 TMAC/s** in a single matmul
- `Wgate` / `Wup` (FFN expand): ~7,100 GMAC/s
- `Wq` / `Wo` (attn proj): ~6,900 GMAC/s

Comparison to production fp16 inference (llama.cpp Q8_0 on same RTX 3090):

| Workload | tokens/s | per-token ms |
|---|---:|---:|
| llama.cpp Q8_0 (fp16 tensor cores) | 43.5 | 23.0 |
| ter sim INT8 TC (M=1, generation) | 153 | 6.55 |
| ter sim INT8 TC (M=64, prefill batch) | 5,556 | 0.18 |

**The ternary substrate on INT8 tensor cores BEATS production fp16 LLM
inference by ~3.5x (M=1) to ~127x (M=64 batch).** The win source: Ampere
INT8 peak is 8x its fp16 peak (285 vs 35 TOPS), and the ternary data layout
maps natively to int8 -- no dequant step, no precision loss going through
fp16. This is H3 substrate-data alignment validated empirically on real
silicon.

The matmul fabric runs at ~2.4% of the INT8 TC theoretical peak at M=64
(6,892 / 285,000). Larger batch sizes (M >= 256, typical prefill) and
better matmul tile shape selection would push closer to peak.

## Updated implications

- **Op-count parity** (1.002x lane-MACs vs fp16): confirmed (sim metric)
- **Wall-clock parity at matmul fabric**: confirmed (CUDA sgemm 6.87 ms vs llama.cpp ~23 ms full forward)
- **Substrate-data alignment win on existing silicon** (H3): confirmed via INT8 tensor cores, ~127x speedup over fp16 production at M=64
- **Energy per op** (J/op for ternary CMOS vs binary CMOS): still requires FPGA prototype

For the silicon-investment argument: the throughput case is now defensible
without custom silicon -- INT8 tensor cores already give the substrate-data
alignment win. The remaining argument for custom ternary silicon hinges on:
(a) energy per op vs INT8 silicon, and (b) memory bandwidth gains from
1.58 bits/weight packing that INT8 can't access. Both still need FPGA-level
measurements to justify.

## End-to-end forward (the honest comparison)

`cuda/ter_cuda_forward.cu` implements the FULL Llama 1B forward in CUDA --
INT8 tensor core matmuls plus custom kernels for RMSNorm, RoPE, attention
(Q@K^T + softmax + AV), SiLU, residuals, KV cache, and argmax sampling.
Random weights (throughput skeleton). Times N tokens end-to-end.

| Setup | tokens/s | ms/token |
|---|---:|---:|
| **ter sim INT8 TC, full forward, RTX 3090** | **14.7** | **68.2** |
| llama.cpp Q8_0 fp16-TC, full forward, RTX 3090 | 130 | 7.7 |

**Result: the naive ternary INT8 implementation runs ~9x SLOWER than
llama.cpp Q8_0 end-to-end.** This invalidates the previous "3.5x faster"
matmul-fabric-only claim as a useful comparison.

Why ter sim is slow despite INT8 TC having 8x higher peak TOPS than fp16:
1. D2H sync points: every `quant_int8_k` blocks on `cudaMemcpy` of the
   scale to host (~12 per layer x 16 layers = ~200 sync points per token)
2. Kernel launch overhead: ~50 launches per layer x 16 = 800 launches per
   token; each ~5-10 us overhead = several ms wasted on launch alone
3. Naive attention kernel: 1 block per head, scalar inner loops, no tensor
   core utilization
4. cuBLAS GEMM at M=1 has high relative overhead; INT8 TC peak needs M>=64
5. No fusion: rmsnorm + quantize + matmul = 3 separate kernels; llama.cpp
   fuses these in custom CUDA kernels

**Honest paper claim:**

- Op-count parity (1.002x lane-MACs vs fp16) is real and substrate-level.
- Substrate-data alignment with INT8 silicon is real (kernels run, INT8 TC
  is used, tokens generate).
- Throughput parity vs production fp16 inference is NOT validated. A
  production-grade ternary INT8 engine (kernel fusion, persistent kernels,
  optimized attention layout, fused quant+matmul) could plausibly reach
  parity or 1.5-2x advantage, but that is engineering work the paper does
  not deliver, not an inherent architectural property.

The paper-defensible claim is: "Substrate-data alignment removes the
multiply step from the matmul fabric (validated); harvesting that into
production wall-clock requires a production-quality INT8 inference engine
(future work)."

The substrate vs FPGA vs custom silicon argument now reduces to: are the
remaining wins (sub-byte memory bandwidth from 1.58 bits/weight, custom
ALU energy/op savings) worth the engineering and silicon investment, given
that even with naive implementation we already pay heavy overhead?

## Packed sub-byte memory layout (the architectural win)

`cuda/ter_cuda_packed.cu` benchmarks ternary weights packed at 4 trits per
byte (2 bits each, mapping `0->0, 1->+1, 2->-1, 3->0`). This is the BitNet
1.58-bits-per-weight memory layout. Compared against cuBLAS sgemm fp32 and
cuBLAS INT8 TC at the same Llama 1B matmul shapes (M=1).

### Memory footprint (the real architectural win)

Total weights per Llama 1B forward (matmul fabric):

| Layout | MB | bytes/elem | vs fp16 |
|---|---:|---:|---:|
| **packed (1.58 b)** | **294.6** | **0.20** | **8.0x less** |
| INT8 | 1178.5 | 1.0 | 2.0x less |
| Q8_0 (incl. scales) | 1325.8 | ~1.13 | 1.78x less |
| fp16 | 2357.0 | 2.0 | -- |

**Packed ternary uses ~5x less memory than Q8_0.** In memory-bound regimes
(single-token generation, edge inference) this would translate to a ~5x
speedup if the kernel is bandwidth-optimal.

### Throughput at M=1 (naive kernel does NOT yet harvest the win)

| Backend | ms / forward | GMAC/s |
|---|---:|---:|
| packed-v4 (1 thread / 4 cols, naive) | 705 | 1.8 |
| packed-v16 (uint32 loads, 16 cols) | 831 | 1.5 |
| cuBLAS sgemm fp32 (sgemv) | 86 | 14.4 |
| cuBLAS INT8 TC | 31 | 39.6 |

Naive packed kernel is **22x slower than cuBLAS INT8 TC**. Implied bandwidth
from packed kernel: 0.4 GB/s of 936 GB/s peak = 0.04% utilization. The
kernel is compute-bound by its naive structure (one thread per column,
serial K-loop), not memory-bound.

### Interpretation

The architectural win (5x memory, 8x vs fp16) is real and documented. The
wall-clock realization requires a custom CUDA kernel comparable to
`bitnet.cpp/TL2` (the same kernel-tuning step that hit the codegen wall
in the F12 BitNet attempt). cuBLAS INT8 TC sets the competitive ceiling
reachable without writing world-class custom kernels.

Notable exception: at the lm_head shape (K=2048, N=128256), packed-v4 runs
at 81 GMAC/s -- higher than sgemm (24) and comparable to INT8 TC (70).
With N large enough, parallel threads saturate and the per-thread K-loop
ceases to dominate. Smaller projections (Wq/Wk/Wv) suffer in the naive
kernel because each thread serially scans 2048 K-elements.

### What's left (architectural exploration enabled by CUDA)

The CUDA backend's value is iteration speed, not production competition.
With it, these architectural experiments become tractable:
- Shared-memory tiled packed kernel (close gap to INT8 TC; validates the
  win is architectural not "thanks to tensor cores")
- Format A end-to-end as a first-class number type (no round-trip noise)
- Trit-native attention (Q@K^T in ternary primitives)
- Dense architectural sweeps (n_trits x tile_K x format) on Llama 1B
- Quality preservation per-layer studies

Each of these is a paper-grade experiment that requires the simulator to
run in seconds, not minutes.
